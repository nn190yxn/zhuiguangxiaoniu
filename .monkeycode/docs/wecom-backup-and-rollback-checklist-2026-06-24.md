# 企业微信数据库变更备份与回滚清单

日期：2026-06-24

## 1. 目的

- 在执行企业微信数据库变更前，先完成结构备份、数据备份和执行前确认。
- 一旦数据库变更影响现有网站、H5 或原微信小程序链路，立即按本清单回滚或止损。

## 2. 执行前备份

### 2.1 结构备份

至少备份以下对象：

1. `staffs`
2. `mini_reminder_rules`
3. `mini_reminder_jobs`
4. `mini_user_notifications`
5. `policy_notifications`
6. `wp_users`

### 2.2 数据备份

至少备份以下数据范围：

1. `staffs` 全表
2. 最近 7 天 `mini_reminder_jobs`
3. 最近 7 天 `mini_user_notifications`
4. 最近 7 天 `policy_notifications`

### 2.3 配置备份

执行前记录：

1. 当前 `api/config.php` 配置来源
2. 当前 `WECOM_*` 是否为空
3. 当前 crontab 完整内容

## 3. 执行前确认

执行数据库变更前，逐项确认：

1. 现有网站和 H5 当前可正常使用。
2. 已完成 `wecom-existing-site-regression-checklist-2026-06-24.md` 的变更前基线确认。
3. 已准备好执行后第一时间回归的账号和页面。
4. 已确认本轮只执行增字段、增表，不做删字段、删表、删数据。

## 4. 回滚触发条件

出现以下任一情况，立即停止继续企业微信联调：

1. H5 账号密码登录失败。
2. 原微信小程序 `wxlogin` 或 `wxbind` 异常。
3. 首页、学习、制度、通知、工作量任一核心链路出现新增 500 错误。
4. 原提醒 worker 无法执行或站内通知异常。
5. `staffs` 查询、后台员工页、学习后台、工作量后台出现回归异常。

## 5. 回滚顺序

### 5.1 立即止损

1. 保持 `WECOM_ENABLED=0`。
2. 不挂企业微信同步 worker。
3. 不挂企业微信消息相关 cron。

### 5.2 如已补配置

1. 先关闭企业微信开关。
2. 再复核 `/api/wecom/status.php` 应回到不可用状态。

### 5.3 如已挂 worker

1. 先停通讯录同步 worker。
2. 再停提醒 worker。
3. 再复核现有站内提醒链路。

### 5.4 数据层处理口径

1. 本轮字段和表属于兼容性新增，优先通过停配置、停 worker 止损。
2. 如现网站与 H5 恢复正常，保留新增字段和表也可接受。
3. 只有在确认新增字段或表本身导致现网异常时，才进入更高等级人工处理。

## 6. 回滚后复核

回滚或止损后，立即复核：

1. H5 账号密码登录。
2. 原微信小程序登录与绑定。
3. 首页、学习、制度、通知、工作量。
4. 原提醒 worker。
5. 员工后台、工作量后台、学习后台。

## 7. 回填要求

每次触发止损或回滚后，立即回填：

1. `wecom-gap-tracking-2026-06-24.md`
2. `wecom-mini-program-master-plan-2026-06-23.md`

回填内容至少包含：

1. 触发时间
2. 触发原因
3. 止损动作
4. 回滚后结果
5. 是否允许继续下一轮执行
