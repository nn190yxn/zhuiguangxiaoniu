# 企业微信专用版小程序改造总计划

日期：2026-06-23

## 1. 背景与目标

- 当前项目已经完成微信小程序主链路收口，核心页面包括登录、协议、首页、学习中心、制度中心、知识卡、通知、工作量日报、我的、通关、销售演练、考试。
- 当前版本已经具备账号密码登录、微信绑定、工作量提醒、学习与考试等内部业务能力。
- 业务方向已确认转向企业微信内部使用场景，目标是将当前员工内网小程序升级为企业微信专用版，统一从企业微信入口访问，并逐步接入企业微信成员、部门、消息、客户联系和数据分析能力。

本次改造的最终目标：

1. 员工统一从企业微信工作台进入应用。
2. 身份体系统一到企业微信成员身份，减少账号密码和额外绑定成本。
3. 保留并复用现有业务模块，重点改造登录、成员映射、提醒、统计和后台同步能力。
4. 建成“企业微信成员体系 + 小程序业务系统 + 客户分析中心 + 企业微信消息提醒”的完整闭环。

## 3. 总体原则

1. 业务页面尽量复用，减少重复建设。
2. 企业微信负责身份、通讯录、消息、企业原生能力接入。
3. 现有后端负责业务数据、权限映射、统计加工和审计日志。
4. 所有统计能力优先走“后端同步 + 本地分析 + 小程序展示”，避免在小程序端直接拼装复杂数据。
5. 所有上线动作采用分阶段灰度，先测试部门，再小范围门店，再全员。

## 3.1 硬性开发要求

以下要求属于本项目改造的强制门槛，所有开发、联调、上线、回滚都必须执行：

1. 所有新能力必须先补文档，再进开发。
2. 所有接口改造必须明确输入、输出、鉴权、失败码、超时策略、重试策略。
3. 所有数据同步能力必须明确源头系统、同步频率、幂等策略、去重规则、失败补偿。
4. 所有权限能力必须明确谁能看、谁能发、谁能跳、谁能管理。
5. 所有对现网有影响的改造都必须先有备份、再有灰度、最后全量。
6. 所有上线都必须带回测清单和回滚开关。
7. 所有关键链路都必须带日志、告警和审计记录。

## 3.2 数据要求

### 3.2.1 数据源要求

必须先定义每类数据的唯一真实来源：

- 企业成员与部门：企业微信通讯录为主源。
- 企业客户、客户标签、客户群关系：企业微信客户联系数据为主源。
- 员工角色、门店归属、业务权限：本地 `staffs` 和现有业务后台为主源，企业微信同步结果用于映射和校验。
- 学习、制度、工作量、考试、演练业务数据：现有业务数据库为主源。
- 应用消息发送记录、同步日志、分析快照：本地新建日志表和统计表为主源。

### 3.2.2 数据质量要求

1. 每张新增业务表必须定义主键、唯一键、更新时间、来源标识。
2. 每个同步表必须定义“最后同步时间”“同步批次号”“数据来源版本”。
3. 客户、成员、标签同步必须支持幂等写入，禁止重复插入脏数据。
4. 所有统计报表必须明确统计口径、时间口径、去重口径、归属口径。
5. 所有分析结果都必须可回溯到原始同步记录。

### 3.2.3 数据安全要求

1. 不在前端存储企业微信敏感配置。
2. Access Token、应用密钥、企业微信配置仅保留在服务端安全配置中。
3. 所有日志默认脱敏，手机号、客户敏感字段、员工认证标识不得明文全量输出。
4. 导出的统计结果如含客户明细，必须经过权限控制。

## 3.3 链路要求

### 3.3.1 登录链路要求

标准链路：

1. 企业微信入口打开小程序。
2. 小程序识别企业微信上下文。
3. 服务端获取企业成员身份。
4. 服务端做 `wecom_userid -> staffs -> role -> permission` 映射。
5. 已映射员工直接进入首页。
6. 未映射员工进入首次绑定页。
7. 绑定完成后重新生成登录态并进入首页。

要求：

- 登录链路单次平均完成时间目标小于 3 秒。
- 任一节点失败都必须返回明确错误，不允许只给模糊超时。
- 管理员兜底登录链路必须全程可用。

### 3.3.2 消息提醒链路要求

标准链路：

1. 定时任务扫描待提醒对象。
2. 按门店、角色、状态生成消息任务。
3. 消息任务写入本地日志表。
4. 调用企业微信应用消息接口发送。
5. 发送结果写回日志表。
6. 用户点击消息进入小程序对应页面。
7. 小程序页面展示与消息内容口径一致。

要求：

- 消息生成、发送、回写三段必须可追踪。
- 失败任务必须支持自动重试和人工补发。
- 同一提醒在同一时间窗口内禁止重复轰炸。

### 3.3.3 客户分析链路要求

标准链路：

1. 拉取企业微信客户联系数据。
2. 清洗客户、标签、成员、门店映射。
3. 写入本地客户分析表。
4. 生成日统计、小时统计、客户分层结果。
5. 小程序读取分析接口展示看板。

要求：

- 原始同步表、标准化表、聚合表三层分离。
- 原始数据层禁止直接被前端读取。
- 统计接口只读聚合结果，避免前端直接打重查询。

## 3.4 逻辑要求

### 3.4.1 身份与权限逻辑

1. 企业微信成员身份优先于普通小程序身份。
2. 一个企业微信成员只能映射一个有效员工档案。
3. 一个员工档案只能绑定一个有效 `wecom_userid`。
4. 停用员工、离职员工、超范围员工登录后必须拦截。
5. 门店、角色、总部权限继续以现有业务权限为准，企业微信数据仅作为身份和组织来源。
6. 网站手机号登录与企业微信登录最终必须统一归属到同一条 `staffs` 员工档案。
7. 统一映射主链固定为 `wp_users.ID -> staffs.user_id <- staffs.wecom_userid`。
8. 所有学习、通知、工作量、考试、通关、演练数据都必须以同一员工档案归属作为判定基准。

### 3.4.1.1 账号主数据统一规则

1. `staffs` 是员工主数据表。
2. 网站手机号登录继续保留在 `wp_users` 账号层。
3. 企业微信成员身份继续保留在 `wecom_userid` 身份层。
4. 业务系统统一通过 `staffs.id` 识别员工归属。
5. 历史仅挂 `user_id` 的业务表必须完成影响审计，确认是否需要补 `staff_id` 或补反查链路。
6. 第二阶段开始前必须出一份账号统一性审计清单，至少覆盖：
- `staffs.user_id` 空值
- 重复手机号
- 重复 `wecom_userid`
- 网站手机号账号与企业微信手机号不一致员工
- 仅依赖 `user_id` 归属的业务表分布

### 3.4.2 客户分析逻辑

1. 客户标签统计必须按客户唯一标识去重。
2. 客户归属必须明确是按企业微信归属成员，还是按本地门店映射规则二次归属。
3. 沉默客户、高意向客户、待跟进客户、流失风险客户必须先冻结业务口径，再开发。
4. 所有趋势统计必须明确时间粒度：按天、按周、按月。

### 3.4.3 放行逻辑

1. 已映射且已完成必要配置的员工重新登录直接进首页。
2. 首次绑定页只在未映射时出现。
3. 首次提醒引导页只在确实未满足提醒配置条件时出现。
4. 全部完成设置后，后续登录不重复出现中间关卡页。

1. 开发前记录当前 `main` 基线提交号。

4. 备份关键配置文件、环境变量说明、定时任务配置。

### 3.5.2 数据备份要求

1. 对 `staffs`、客户分析新表、提醒相关表、权限相关表做上线前快照备份。
2. 执行任何不可逆数据迁移前，必须先在测试库完整演练。
3. 所有数据迁移脚本必须支持回滚或补偿。

### 3.5.3 接口与配置基线

1. 记录现有登录链路接口响应样本。
2. 记录现有提醒链路接口响应样本。
3. 记录现有首页、通知、待办、工作量关键页面访问结果。
4. 企业微信应用权限变更前必须导出当前配置截图或文本基线。

## 3.6 开发过程要求

1. 每完成一个模块必须立即完成模块级静态检查和联调记录。
2. 所有数据库脚本、同步脚本、定时任务脚本必须单独审阅。
3. 所有接口在开发阶段都要补最少一组成功样例和一组失败样例。
4. 所有第三方调用必须设置连接超时、总超时、失败日志和降级文案。
5. 涉及企业微信外部接口的功能必须定义限流策略和失败补偿。

## 3.7 开发后回测要求

### 3.7.1 回测范围

每次发版后至少回测以下链路：

1. 企业微信入口登录
2. 员工首次绑定
3. 已绑定员工再次登录
4. 工作量提醒发送与进入页面
5. 制度通知进入页面
6. 首页待办与通知列表
7. 工作量日报提交与后台可见
8. 学习、知识卡、考试、演练关键页面访问
9. 客户分析统计页与明细页

### 3.7.2 数据回测要求

1. 成员同步结果与企业微信后台抽样对比。
2. 客户总量与企业微信客户后台抽样对比。
3. 标签统计与企业微信标签维度抽样对比。
4. 门店归属和员工归属抽样对比。
5. 提醒发送日志与用户实际收到情况抽样对比。

### 3.7.3 回测判定标准

- 核心链路通过率必须达到 100%。
- 核心报表口径误差必须小于 1%。
- 企业微信消息发送失败率必须低于 1%，超过阈值立即阻断上线。

## 3.8 发布门禁要求

上线前必须同时满足：

1. 数据库脚本在测试环境已完成演练。
2. 企业微信权限和可见范围已由管理员确认。
3. 关键链路回测记录已完整留档。
4. 回滚开关已准备完成。
5. 同步任务、消息任务、日志监控已部署。
6. 上线窗口、责任人、观察窗口、回滚负责人已明确。

## 4. 现状评估

### 4.1 已具备能力

- 小程序主业务模块已经可用。
- 登录后绑定微信、提醒授权、首页放行逻辑已经打通。
- 工作量、学习、知识卡、演练、考试已确认与后台统一后端落库。
- 已具备站内提醒链路和部分手机订阅提醒基础能力。
- 已新增工作量提醒任务接口、订阅授权接口、任务查询接口和 worker 骨架。

### 4.2 当前与企业微信专用版的主要差距

- 登录体系仍以普通小程序账号密码登录为主。
- 员工身份仍依赖本地账号和微信绑定状态。
- 提醒能力仍以小程序授权为主，未切到企业微信应用消息。
- 成员、部门、角色、门店关系未与企业微信通讯录统一。
- 客户联系数据尚未进入本地分析体系。

## 4.3 现有模块影响评估

企业微信专用版改造期间，必须明确区分“内容层复用”和“访问控制层改造”，避免误把现有业务模块当成独立重写对象。

### 4.3.1 可直接复用的业务内容层

以下模块的内容资产、业务数据和核心页面结构原则上继续复用：

- 知识库
- 制度中心
- 学习中心
- 知识卡
- 销售演练
- 通关
- 考试
- 工作量日报
- 通知内容

要求：

1. 不因企业微信改造重做原有内容数据结构。
2. 不在无必要情况下迁移历史业务数据。
3. 页面内容和业务能力以现有线上可用逻辑为基线。

### 4.3.2 必然受影响的访问控制层

以下能力在企业微信专用版中必须改造：

- 登录入口
- 员工身份识别
- 成员映射关系
- 页面放行逻辑
- 角色与门店权限判断
- 消息提醒触达方式
- 统计归属口径

要求：

1. 所有存量业务模块都必须重新经过企业微信身份链路验证。
2. 所有角色判断都必须核对企业微信成员身份映射后是否仍然正确。
3. 所有提醒和待办都必须确认在企业微信消息接入后口径一致。

### 4.3.3 重点防回归模块

以下模块在改造中属于高风险模块，必须纳入重点回归清单：

1. 知识库
2. 制度中心
3. 销售演练
4. 通关
5. 考试
6. 工作量日报
7. 通知与待办

### 4.3.4 模块迁移要求

#### 知识库

- 继续复用现有知识内容、分类、完成态记录。
- 必须验证企业微信登录后是否还能正确读取列表、详情、完成状态。
- 必须验证知识完成记录是否仍然正确归属到当前员工。

#### 制度中心

- 继续复用现有制度列表、制度详情、确认阅读记录。
- 必须验证制度通知、确认状态、未读状态在企业微信登录后是否保持一致。
- 必须验证门店和总部角色看到的制度范围是否正确。

#### 学习中心

- 继续复用课程、章节、学习进度。
- 必须验证企业微信成员映射后课程进度是否仍能正确读取和写入。

#### 销售演练

- 继续复用演练场景、脚本、训练记录、语音输入能力。
- 必须验证企业微信登录后训练记录是否仍然正确归属员工。
- 必须验证语音能力、脚本复制、记录保存不受身份链路改造影响。

#### 通关与考试

- 继续复用题库、答题记录、成绩统计。
- 必须验证企业微信登录后答题、交卷、历史进度和成绩归属是否正确。

#### 工作量日报

- 继续复用日报模板、日报记录、凭证、后台汇总逻辑。
- 必须验证企业微信身份登录后门店归属、角色归属、汇总归属是否正确。
- 必须验证员工提交后后台仍能读取同一份数据。

#### 通知与待办

- 继续复用现有站内通知和待办聚合逻辑。
- 企业微信消息接入后，必须确认站内通知、待办列表、企业微信消息三者口径一致。

### 4.3.5 模块影响验收要求

每个存量模块都必须输出以下结论：

1. 是否直接复用。
2. 是否需要轻改。
3. 是否需要重构身份或权限逻辑。
4. 是否需要迁移历史数据。
5. 是否需要专项回归测试。

未完成以上五项评估的模块，不允许直接进入开发实施。

## 5. 改造总目标图

### 5.1 最终运行链路

1. 员工从企业微信工作台进入小程序。
2. 小程序识别当前企业成员身份。
3. 后端用企业微信 `userid` 映射到本地员工档案。
4. 已映射员工直接进入首页。
5. 未映射员工进入一次性绑定确认页。
6. 业务页面继续使用现有模块。
7. 工作量、制度、学习、客户跟进提醒通过企业微信应用消息推送。
8. 客户标签、客户分组、客户状态等数据由企业微信客户联系接口同步到本地分析库，再在小程序中展示分析看板。

## 6. 分阶段实施计划

## 6.1 第一阶段：企业微信基础接入

目标：完成企业微信内部入口、成员身份识别、员工映射和直接登录。

## 14. 回滚方案

1. 保留现有账号密码登录链路作为兜底
2. 企业微信登录通过配置开关控制
3. 企业微信消息通过配置开关控制
4. 客户分析中心可独立关闭，不影响基础业务
5. 如企业微信接入异常，可快速回退到当前普通小程序登录与业务链路

### 回滚触发条件

1. 企业微信登录大面积失败，员工无法正常进入系统。
2. 企业微信消息大面积失败，影响工作量、制度、学习关键提醒送达。
3. 企业微信身份映射异常导致员工数据归属错误。
4. 企业微信消息点击跳转错误导致关键业务页面不可达。
5. 线上同步异常持续扩大，导致门店、岗位、成员状态大量错配。

### 回滚执行口径

1. 优先关闭企业微信登录入口，保留账号密码链路和现有业务链路。
2. 关闭企业微信消息发送开关，保留站内提醒主链路。
3. 停止企业微信相关 worker 和 cron，避免继续写入异常数据。
4. 通过后台审计和日志确认异常影响范围，再决定是否恢复同步或重新灰度。

### 回滚后保留项

1. 已有后台审计页面、日志表和清单导出能力继续保留，用于排障和复盘。
2. 已完成的主数据清理结果继续保留，不回退已确认正确的员工映射。
3. 客户分析中心维持未上线状态，不影响基础业务回退。
